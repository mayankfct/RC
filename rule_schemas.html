Schemas · Final
Rule schemas for the
detection module
The contracts a rule author must satisfy and the agent's compile step enforces. Sigma is YAML on the wire; YARA is the standard language constrained to the agent's allowed feature set; behavioral is YAML compiled to MessagePack; signatures live in a Bloom + side index. Every rule type produces the same AlertRecord output.

Why these schemas, why this strict

A rule that compiles to slightly the wrong shape doesn't fail loudly — it silently never matches. The schemas here are deliberately narrow: required fields are required, modifiers are an explicit allowlist, names are exact-match against the agent's NormalizedEvent struct. CI rejects anything outside the schema before it reaches an agent. This is not bureaucracy — it's the only way to keep the rule corpus debuggable as it grows past a few hundred rules.

Schema · Sigma
Sigma rule schema
YAML rules compiled at agent load to CompiledMatcher structs. Stateless: one event in, match-or-not out. The schema below is the EDR-internal Sigma dialect — strict subset of upstream Sigma's modifier list, augmented with EDR-specific fields (auto_action, dedup_window_seconds, etc.). Universal fields from the Overview page are required in addition to what's listed here.

Sigma-specific required fields

FieldTypeRequiredNotes
event_typeenumyesPre-filter. MUST match an entry in the NormalizedEvent enum (see Cross-cutting → NormalizedEvent). Rule never evaluates on other event types.
selectionmap | named-groupyesPositive condition. Either a single map (default condition: selection) or multiple selection_* blocks combined via the condition field.
filtermapoptKnown-good carve-out. If filter matches, alert is suppressed even when selection matched.
conditionstringoptRequired only when multiple selection_* blocks exist. Examples: 1 of selection_*, selection or selection_combo.

Boolean semantics — exactly what gets evaluated

# Within a selection block:
selection:
  field_a: x         # field_a == x  AND
  field_b: y         # field_b == y  AND
  field_c|any: [...] # field_c in list (any-of)
                     # Everything in the block is ANDed.

# With multiple selection_* blocks:
selection_a: { ... }
selection_b: { ... }
condition: 1 of selection_*    # OR  (default if condition omitted)
condition: all of selection_*  # AND
condition: selection_a or selection_b  # explicit boolean

# Filter is evaluated AFTER selection.
# If filter matches, the rule is suppressed.
filter:
  field_a: known_good   # suppress when this matches

Schema · YARA
YARA rule schema
Standard YARA language constrained to the yara-x runtime's feature set, with a required meta block whose keys are checked by CI. Authors write .yar source; the build pipeline compiles each file with yarac into a single packed .yrc bundle which the agent mmaps.

Meta block — required keys

YARA's meta block is conventionally free-form. Our CI requires the keys below — this is what makes the meta block parseable into the AlertRecord without per-rule special cases.

Meta keyTypeRequiredNotes
idstringyesFormat Y-NNN. Same value as filename prefix.
familystringyesMalware family or technique class. Camel/PascalCase.
descriptionstringyesWhat it detects, briefly.
authorstringyesSame enforcement as Sigma.
datestringyesYYYY-MM-DD.
severityenumyescritical · high · medium · low.
mitrestringyesComma-separated technique IDs in a single string (YARA meta values must be scalar).
referencestringoptURL.

Rule identifier convention

YARA identifiers cannot contain hyphens. Rule names follow Y_NNN_FamilyName (underscores). Filename uses hyphens: Y-NNN_lowercase_family.yar.

Required structural anchors

Every rule must include at least ONE of the structural anchors below in its condition. Pure-string rules without anchors are rejected by CI. Anchors keep scan time bounded and make rules robust against false positives on unrelated file types.

AnchorPurpose
uint16(0) == 0x5A4DPE file (MZ header). Use for any rule targeting Windows binaries.
uint32(0) == 0x464c457fELF file. Use for Linux samples (when relevant).
filesize < NHard cap. Required on every rule. Prevents pathological large-file scans.
filesize > MLower bound. Use to skip stub files / dropper shells.

Allowed yara-x features

FeatureAllowedNotes
ascii / wide / nocase string modifiers✓Standard usage.
Hex byte patterns with wildcards✓{ 8B 45 ?? 33 ?? }
Regex strings✓Bounded — <200 chars per pattern.
math.entropy()✓For packer / encryption detection. Use sparingly — cost is per-scan, not per-rule.
pe module (imports / exports / sections)✓Use for header-based pivots.
2 of, 3 of, any of, all of✓Combinator conditions — required. See below.
cuckoo module✗Sandbox-only, not on agent.
elf, magic modulescase-by-caseOff by default in yara-x. Confirm with engineering before use.

Combination requirement

Single-string rules (condition: $a) are rejected by CI. Every condition must combine at least two indicators OR pair one strong indicator with a structural anchor. This is the difference between a rule that fires on every README mentioning PowerShell and a rule that fires on actual malware.

AcceptableWhy
2 of ($brand_*)Two distinct family-marker strings — robust against incidental matches.
$marker and 2 of ($cfg_*)Brand string + configuration markers — high specificity.
$loader_prologue and ($api_a or $api_b)Code shape + API indicator.
$rare_pdb_stringSingle string acceptable IF the string is uniquely diagnostic (PDB path with developer username, banner). Justify in description.

Y-002 — full schema example
/*
 * Y-002  Mimikatz
 *
 * Detects Mimikatz binaries by command strings, banners, and
 * privilege-escalation flags. Strings chosen to survive recompilation.
 */
rule Y_002_Mimikatz
{
    meta:
        id          = "Y-002"
        family      = "Mimikatz"
        description = "Mimikatz credential dumping tool"
        author      = "SOC-1"
        date        = "2026-05-06"
        severity    = "critical"
        mitre       = "T1003.001"

    strings:
        $cmd_sekurlsa  = "sekurlsa::logonpasswords" ascii wide
        $cmd_lsadump   = "lsadump::sam" ascii wide
        $cmd_privilege = "privilege::debug" ascii wide
        $banner_a      = "gentilkiwi" ascii wide
        $banner_b      = "mimikatz" ascii wide nocase

    condition:

}

Schema · Behavioral
Behavioral rule schema
Stateful detection: chain patterns across multiple events, sliding-window counters per PID, sibling bursts under one parent. Authored as YAML, compiled at build time to MessagePack for compact runtime parsing.

Three rule types under one schema

typeWhat it doesExample
chainSequence of events matching ordered stages within a windowCP-001: Office → shell → download
sibling_burstN distinct processes from same parent within windowCP-002: 3 recon tools in 30s
counterPer-PID sliding-window thresholds on event rates / sumsB-RANSOM-001: write-rate + new-extension count

Common required fields (in addition to universal)

typeenumyeschain · sibling_burst · counter. Determines which other fields are required below.
window_secondsintyesRolling time window in seconds. No defaults — must be explicit.
filtermapoptSame semantics as Sigma. Counter rules effectively require this — rate-based detection without a filter floods on dev tooling.
enabledbooloptDefault true. Set false to ship the rule but not evaluate (e.g. waiting on a provider — see B-INJECT-001).

Schema · Signatures
Hash signature schema
Three artifacts: the source feeds config (where hashes come from), the server-side hashes table (single source of truth), and the agent-side Bloom filter (compact membership test). Each agent gets a Bloom — never raw hashes. The cloud-lookup verdict structure is the response shape for Bloom hits.

Architecture recap

SOURCE FEEDS  ──nightly pull──>  server hashes table  ──nightly build──>  *.bloom (~1.2MB)
   (TOML)                          (Postgres / OpenSearch)                   ──>  agent

AGENT FileClose  ──hash──>  bloom.contains(sha256)?
                                NO  → done (99% of files, no network)
                                YES → cloud_lookup(sha256) → Verdict
                                          Malicious  → quarantine + alert
                                          Clean      → cache, do nothing
                                          Unknown    → cache TTL, no alert

3. Agent-side .bloom file format

Compact, mmap-friendly binary file. The agent never parses it as data — it mmaps and indexes.

bloom file layout
// File: malshare.bloom (or whatever feed name)
// All values little-endian.

struct BloomFile {
    magic:           [u8; 4],     // "EDRB"
    format_version:  u16,           // 1
    hash_count:      u8,            // k — number of hash funcs (typ 7)
    reserved:        u8,
    n_items:         u64,           // approximate count of hashes
    bit_count:       u64,           // m — size of bit array in bits
    fp_rate_x1e6:    u32,           // FP rate × 1,000,000 (e.g. 100 = 0.0001)
    built_at_utc:    u64,           // Unix epoch seconds
    feed_name_len:   u16,
    feed_name:       [u8; feed_name_len],
    bit_array:       [u8; ceil(bit_count / 8)],
    sha256_self:     [u8; 32],    // hash of all preceding bytes
    signature:       [u8; 64],    // Ed25519 sig over sha256_self
}

Why signed: an attacker who can replace active\signatures\malshare.bloom on disk could blank it out (silencing detection) or stuff it (causing storm of false-positive cloud lookups). Ed25519 sig with the public key embedded in the agent binary at compile time stops both.

4. Verdict response (cloud lookup)

What the cloud-lookup gRPC returns when an agent sees a Bloom HIT and asks the server.

VerdictLookup gRPC response
message VerdictResponse {
    string sha256       = 1;       // echoes the request
    Verdict verdict     = 2;
    string family       = 3;       // "" if unknown
    string first_seen   = 4;       // ISO 8601, "" if unknown
    int32  ttl_seconds  = 5;       // agent caches result for this long
    repeated string tags = 6;       // free-form
}

enum Verdict {
    UNKNOWN     = 0;          // Bloom FP, or genuinely unknown — short TTL (60s)
    CLEAN       = 1;          // allowlist hit — long TTL (24h)
    SUSPICIOUS  = 2;          // alert but no auto-quarantine
    MALICIOUS   = 3;          // alert + quarantine + suspend chain
}

5. Allowlist (separate from blocklist)

Known-good hashes shipped as a separate Bloom — allowlist.bloom. Same file format. Used to suppress alerts on hashes that DO match a YARA rule (false positive in YARA) but are known-clean. Lookup order on FileClose:

1. Compute sha256
2. allowlist.bloom.contains(sha256)?  YES → skip ALL engines for this file
3. blocklist.bloom.contains(sha256)?  YES → cloud lookup → quarantine on Malicious
4. Run YARA scan
5. Run other engines normally
The allowlist Bloom can run at higher FP rate (0.001 = 1 in 1000) without impact — a false-positive on the allowlist just means we skipped a good file's scan, which is acceptable. Blocklist must stay tight (0.0001) because every FP costs a cloud round-trip.

Cross-cutting · MITRE ATT&CK
Tactics & Techniques
Every rule maps to one primary MITRE technique plus zero-or-more secondary techniques. Primary is what the rule's match directly indicates; secondaries are broader attack-chain context. This page is the schema for the mitre: block referenced from every other rule type's universal fields.

Why primary + secondary, not a flat list

Production EDR consoles (CrowdStrike, SentinelOne, Defender, Elastic) all converge on the same pattern: one technique gets the spotlight in alert lists, all techniques get stored for heatmaps and pivoting. If an alert shows up under three tactics simultaneously, it floods the SOC's tactic-filtered queue three times. The rule author picks the primary; the others are context.

Where it showsWhat's displayed
Alert list viewPrimary tactic + technique only ("Execution — PowerShell")
Alert detail viewPrimary at top + all secondaries grouped by tactic
MITRE heatmap dashboardCounts both primary and secondaries — one alert with 3 techniques adds +1 to each cell
"Show similar alerts" pivotPivots on primary by default; clicking a secondary repivots
API filter ?tactic=executionMatches if execution is primary OR a secondary

The block — structure and required fields

mitre block — full schema
mitre:
  primary:                              # REQUIRED — single object
    tactic_id:    TA0002              # MITRE tactic ID
    tactic:       "execution"         # tactic name (lowercase-hyphenated)
    technique_id: T1059.001           # MITRE technique ID (sub-techniques OK)
    technique:    "PowerShell"        # technique name (human readable)

  secondary:                            # OPTIONAL — list of objects
    - tactic_id:    TA0001
      tactic:       "initial-access"
      technique_id: T1566.001
      technique:    "Spearphishing Attachment"
    - tactic_id:    TA0002
      tactic:       "execution"
      technique_id: T1204.002
      technique:    "Malicious File"

Per-entry fields — all four are required

FieldTypeNotes
tactic_idstringAlways TAxxxx format. The 14 MITRE Enterprise tactic IDs.
tacticstringLowercase-hyphenated name (e.g. credential-access). Matches MITRE's canonical short name.
technique_idstringEither Txxxx (top-level) or Txxxx.yyy (sub-technique). Always use the most specific available.
techniquestringThe official MITRE technique name. CI validates against the MITRE catalogue.
Duplicating tactic_id + tactic in every entry is intentional. It means the agent and console never have to do a join against a tactic table to display an alert — everything needed is in the rule itself. The tactic name is also what changes if MITRE renames a tactic; storing both gives a fast rename path.

Heatmap query — counting both

The MITRE heatmap on the dashboard counts every technique cited in every alert. The query is union-over-primary-and-secondaries:

ClickHouse — MITRE heatmap query
SELECT
    technique_id,
    technique,
    tactic_id,
    tactic,
    count() AS hit_count
FROM (
    -- primaries
    SELECT
        mitre_primary_technique_id  AS technique_id,
        mitre_primary_technique     AS technique,
        mitre_primary_tactic_id     AS tactic_id,
        mitre_primary_tactic        AS tactic
    FROM alerts
    WHERE timestamp_utc >= now() - INTERVAL 7 DAY

    UNION ALL

    -- secondaries (array-joined)
    SELECT
        JSONExtractString(s, 'technique_id') AS technique_id,
        JSONExtractString(s, 'technique')    AS technique,
        JSONExtractString(s, 'tactic_id')    AS tactic_id,
        JSONExtractString(s, 'tactic')       AS tactic
    FROM alerts
    ARRAY JOIN JSONExtractArrayRaw(mitre_secondary_json) AS s
    WHERE timestamp_utc >= now() - INTERVAL 7 DAY
)
GROUP BY technique_id, technique, tactic_id, tactic
ORDER BY hit_count DESC;

Maintaining the catalogue

MITRE updates the ATT&CK framework roughly twice a year. New techniques get added; some get deprecated. The CI MITRE validator pulls from https://github.com/mitre/cti (the official STIX 2.x repo) on build, so new techniques become valid automatically. Deprecated techniques generate a warning, not a reject — give SOC-1 a quarter to migrate.

Cross-cutting · FP tuning
FP tuning log
Every filter change to a stable rule (adding a known-good carve-out to suppress a false positive) is recorded in FP_TUNING_LOG.md in the rules repo. The log is the authoritative history of WHY each filter exists, signed off by SOC-1, and required reading before any filter edit. This page is the schema for what goes in that log and the revision field on rules.

Why a dedicated log

Filters accumulate over time. Without a written record, three things break:

Failure modeHow the log prevents it
New authors revert hard-won FP fixes during cleanupEvery filter has a signed-off entry — reverting requires a counter-entry, not a stealth edit
Customer asks "why didn't you alert on X?" and SOC can't answerThe log is searchable by rule ID — find the filter, find the FP report that motivated it
CI soak test catches FP, gets fixed, then regresses 6 months laterThe FP scenario becomes a permanent fixture in tests/fp_corpus/ when the entry is added

The revision field — version vs revision

Every rule carries two version-tracking fields. They serve different audiences.

FieldBumps whenSemantic
versionDetection logic changes — selection, condition, thresholds, or what events the rule looks at"Rule got smarter / different"
revisionOnly filters tuned (FP suppression). Rule fires on the same things; it just suppresses known-good cases more accurately"Rule got quieter"

Separating these matters for two reasons. First, the CI manifest can tell the agent "your rule got noisier from version 4 to 5" vs "your rule got quieter from revision 1 to 2 of version 4". Second, customers who want stable detection behaviour with their own FP tuning can opt to receive only version changes — revision bumps stay on the server side as suggested tunings.

Log entry schema

FieldRequiredNotes
dateyesISO date the filter landed
ruleyesRule ID (R-NNN, Y-NNN, etc.)
filter_addedyesThe literal filter clause, copy-pasteable
suppressesyesOne-line description of the FP case
real_attack_still_caughtyesYes/no + one-line justification. If "no," the filter is too broad and should be tightened.
filter_typeyesnarrow · signer · path · combo · behavioral
authoryesSOC-1 member who signed off
fixture_pathrecommendedPath to the CI fixture that captures this FP — proves the filter works

Filter type policy

TypeWhat it isWhen to use
narrowSpecific name pattern, path, or hash. Targeted carve-out.When the FP is a single tool with a predictable artifact
signerParent or self code-signing identity check. Tight allowlist.When a vendor signs every binary in a workflow
pathDirectory or path prefix.Discouraged. Attackers can drop binaries anywhere. Only use when path is OS-managed.
combonarrow + signer. The recommended default for new filters.Almost always. Generalizes without being too broad.
behavioralAdjust counter threshold or window without changing what's filtered.For counter-based rules where the FP is rate / volume, not identity
Signer-only is never "signed by anyone." Code-signing certs leak and get stolen — a 2022 NVIDIA breach yielded certs that were then used to sign malware. Every signer filter MUST specify a tight allowlist of signer substrings, validated against the approved trusted-vendor list in FP_TUNING_LOG.md.

Trusted-vendor list — adding a vendor

The FP tuning log includes an "approved trusted-vendor list" of signer strings pre-cleared for parent-signer filters. Adding a vendor requires:

StepWhat's verified
1Confirm exact Authenticode signer string from a known-good signed binary on a clean machine
2Vendor has a published security disclosure process
3Vendor has a code-signing key custodianship policy
4PR adds vendor to the table AND to the specific rule's parent_image_signer list, with FP-log entry citing the customer report or soak-test result

Example entry — R-021 Claude Code CLI filter

Markdown table row from FP_TUNING_LOG.md
| 2026-05-11 | R-021 | `target_path|re: '\\claude-[a-z0-9-]+-cwd\.exe$'`  | Claude Code CLI temp helper binaries in %TEMP%  | Yes. Pattern is tool-specific (Anthropic-shipped name) and matches only a binary signed by Anthropic | narrow | SOC-1 |
| 2026-05-11 | R-021 | `parent_image_signer|contains|any: [anthropic, microsoft, jetbrains, github inc, docker inc, hashicorp, cursor ai]` | Signed dev-tool vendor processes writing temp helpers | Yes. Real malware in TEMP comes from unsigned or untrusted-vendor parents (Office macros, browsers, scripts) | signer | SOC-1 |

Reverting a filter

To revert a filter — e.g. a previously-trusted vendor's signing key is compromised — open a counter-entry. Do NOT silently remove the original entry; the audit trail must remain intact.

StepAction
1Counter-entry: same date, "Filter added" column reads REVERTED <original-date>
2Drop the corresponding CI fixture so it stops being asserted as "must NOT alert"
3Bump revision on the rule
4Run the corpus to verify no fresh FPs surface — if they do, the revert needs a replacement filter, not just removal

Delta protocol shape

HeartbeatRequest / RuleDelta
message HeartbeatRequest {
  string agent_id            = 1;
  uint64 current_bundle_version = 2;
  map<string, string> per_type_hashes = 3;     // from manifest
  RuleHealth health           = 4;
}

message HeartbeatResponse {
  uint64 new_bundle_version  = 1;
  repeated RuleDelta deltas    = 2;     // only changed rules
  FullBundleRef full_bundle    = 3;     // set only if delta>30% or agent>5 versions behind
}

message RuleDelta {
  string rule_id    = 1;
  DeltaAction action = 2;     // ADDED · CHANGED · REMOVED
  bytes  content    = 3;     // raw rule bytes (YAML / .yrc / .bloom / .onnx)
  string content_hash = 4;
}

message RuleHealth {
  repeated string failed_rule_ids = 1;
  uint32  dropped_events_per_min = 2;
  uint32  active_rule_count      = 3;
}

Cross-cutting · Process Tree
Process tree & lineage
Every event reaching detection is enriched with parent context — but for behavioral chains, alert investigation, and ML features the agent needs more than just immediate parent. It needs the full ancestry, the descendants, and stable identity that survives PID reuse. This page is the schema for the in-memory process tree, how it gets populated, and the fields it exposes to rules and alerts.

Why a tree, not just parent pointers

Windows reuses PIDs. A short-lived PID 1234 (notepad) can exit, then 30 seconds later PID 1234 (cmd.exe spawned by powershell) gets created. If an alert fires later citing "PID 1234's parent was notepad" the SOC investigation goes off a cliff. Production EDRs solve this by minting a stable per-spawn identifier and persisting full ancestry in a tree the agent owns.

Three things the tree gives you that parent-on-event cannot:

CapabilityWhy it needs the tree
Full chain in alerts (5–10 ancestors deep)Single event only carries direct parent. Chain has to be walked from somewhere.
Behavioral chain rules with parent_matchStage 3 needs to know stage 1's PID was an ancestor — not just direct parent.
Stable identity across PID reuseprocess_uid survives even when Windows hands the PID to a different process later.
Descendant queries (kill the whole chain)Auto-action suspend_chain needs to walk down from a PID, not up.
Process lifetime metrics for ML features"How long has this process been alive" requires start_time kept somewhere.

ProcessNode — the per-process record

Lives in HashMap<u32, ProcessNode> on the agent (keyed by PID). Bounded at 8K entries — eviction policy is "exited > 60s ago + no pending child events". Snapshot serialized to SQLite every state_sync_interval for crash recovery.

ProcessNode — agent in-memory schema
pub struct ProcessNode {
    // === Identity ===
    pub pid:               u32,                  // Windows PID — reused over time
    pub ppid:              u32,                  // parent PID at spawn
    pub process_uid:       Ulid,                 // stable agent-minted ID (UNIQUE per spawn)
    pub parent_uid:        Option<Ulid>,         // parent's process_uid (None if parent unknown)
    pub root_uid:          Ulid,                 // top-of-tree ancestor's uid
    pub session_id:        u32,
    pub logon_id:          u64,
    pub user_sid:          String,
    pub user_name:         String,

    // === Program identity ===
    pub process_name:      String,               // lowercase basename "powershell.exe"
    pub image_path:        String,               // lowercase Win32 path
    pub command_line:      String,               // full, lowercase, truncated at 8KB
    pub image_sha256:      String,               // hash of binary on disk
    pub image_signed:      bool,
    pub image_signer:      String,
    pub integrity_level:   IntegrityLevel,

    // === Lifecycle ===
    pub start_time_ns:     i64,                  // nanos since epoch
    pub stop_time_ns:      Option<i64>,
    pub exit_code:         Option<i32>,
    pub state:             ProcessState,         // Running · Stopped · Suspended · Quarantined

    // === Tree structure ===
    pub children:          SmallVec<[u32; 4]>,    // child PIDs (size hint: 4)
    pub depth:             u8,                   // distance from root_uid

    // === Counters used by behavioral / ML — incremented per event ===
    pub file_writes_total: u32,
    pub net_outbound_total:u32,
    pub child_spawn_total: u32,
    pub distinct_extensions_written: u32,
    pub first_outbound_ip: Option<String>,

    // === Chain pattern state ===
    // bitset of behavioral stages this process has matched
    // — used by chain matcher to short-circuit on later events
    pub matched_stages:    u64,                  // bit N = matched stage N
    pub capture_bag:       SmallMap<String, String, 4>,  // stage captures (e.g. reg_value_data)
}

process_uid — why ULID, not PID

PIDs reuse, ULIDs don't. Every ProcessStart mints a fresh ULID. The ULID becomes the stable identity used in alerts, in the alert console, and in any cross-system correlation. PID stays around for runtime indexing (it's what arrives on every event), but the ULID is the truth.

FieldUsed forPersists across
pidRuntime lookup, ETW correlationProcess lifetime only — recycled by Windows
process_uidAlert correlation, history queriesForever — never reused, never collides
parent_uidStable parent reference in alertsForever — points to a specific spawn
root_uid"Show me everything from this attack"Forever — anchors a whole chain
ULIDs are time-sortable, so WHERE root_uid = X ORDER BY process_uid in ClickHouse / OpenSearch returns the ancestry in spawn order naturally.

Tree population — what happens on each event

EventTree action
ProcessStartMint process_uid. Look up parent by PID. If parent absent (race or out-of-order), enqueue per the EX-C-003 hold-buffer. Set parent_uid + root_uid. Push self into parent's children. Set depth = parent.depth + 1.
ProcessStopSet stop_time_ns, exit_code, state = Stopped. Don't remove yet — wait 60s for late child events. Then evict and remove from parent's children.
FileClose / NetConnect / RegSet etc.Increment relevant counter on the owning process node. Update matched_stages if this event satisfies a chain stage selector.
Suspend / Quarantine actionSet state accordingly so subsequent rules know this PID is already neutralized.

Lookup operations the engines need

Engines and the alert builder don't read ProcessNode fields directly — they call these query functions. Implementation lives in state/process_tree.rs.

process_tree.rs — public API
impl ProcessTree {
    // === Single-process lookups ===
    pub fn get(&self, pid: u32) -> Option<&ProcessNode>;
    pub fn get_by_uid(&self, uid: Ulid) -> Option<&ProcessNode>;

    // === Ancestry walks ===
    // Returns oldest-first (root → self), capped at `depth`.
    pub fn ancestry(&self, pid: u32, depth: u8) -> Vec<ChainEntry>;

    // "Is `ancestor_pid` in `pid`'s ancestry chain?"
    // Used by behavioral parent_match.
    pub fn is_descendant_of(&self, pid: u32, ancestor_pid: u32) -> bool;
    pub fn is_descendant_of_uid(&self, pid: u32, ancestor_uid: Ulid) -> bool;

    // === Descendant walks ===
    // Used by suspend_chain auto-action.
    pub fn descendants(&self, pid: u32) -> Vec<u32>;

    // === Chain stage matching ===
    // True if `pid` (or any ancestor up to `max_depth`) has matched stage `stage_id`
    // within `window_seconds` ago.
    pub fn ancestor_matched_stage(
        &self, pid: u32, stage_id: u8,
        max_depth: u8, window_seconds: u32,
    ) -> Option<Ulid>;    // returns the matching ancestor's uid

    // === Capture access ===
    // Look up a value captured at an earlier stage.
    pub fn get_capture(&self, uid: Ulid, key: &str) -> Option<&str>;
}

ChainEntry — what gets embedded in alerts

Every AlertRecord carries a process_chain field of up to 5 ancestors. ChainEntry is the per-ancestor shape. Sized for backend storage and SOC readability — full ProcessNode would be too large.

How rules reference the tree — by engine

Sigma — direct parent only (cheap)

Sigma's match loop is per-event and stateless. It uses the immediate parent fields already on NormalizedEvent — no tree walk per match (would blow the perf budget). For "grandparent" detection, write a behavioral chain rule instead.

selection:
  parent_image_path|endswith: '\winword.exe'
  process_name|endswith: '\powershell.exe'

Behavioral — full ancestry via parent_match

Chain rules use parent_match: stage_name. The matcher calls ancestor_matched_stage() which walks up to max_depth levels (default 5). This is how CP-001 stitches "office → shell → download" even though they're not direct parent-child at every step (PowerShell could spawn a helper that does the download).

stages:
  - name: stage_office
    process_name|endswith|any: ['\winword.exe', '\excel.exe']

  - name: stage_download
    parent_match: stage_office     # walks ancestry up to max_depth
    parent_match_max_depth: 5         # default — override per stage if needed
    process_name|endswith: '\certutil.exe'

ML — ancestry as feature vector

The feature spec exposes ancestry through dedicated transforms. ML models that benefit most from lineage: process anomaly (parent process is highly predictive), lateral-movement (chain-of-RDP detection), persistence (was-spawned-by-installer-process).

Auto-action on the tree — suspend_chain semantics

A new auto-action on top of suspend_process: suspend_chain walks descendants of the matched PID and suspends them all. Used by ransomware detection (kill the encryption process AND any helpers it spawned) and by chain-rule auto-action (kill the whole 3-stage chain, not just the leaf).

auto_action: suspend_chain — execution
// Pseudocode — response/suspend.rs
fn suspend_chain(tree: &ProcessTree, root_pid: u32) -> SuspendResult {
    let targets = tree.descendants(root_pid);
    let mut suspended = vec![];
    let mut failed = vec![];

    // Suspend leaf-first to prevent re-spawning during walk.
    for pid in targets.iter().rev() {
        match nt_suspend_process(*pid) {
            Ok(_)  => suspended.push(*pid),
            Err(e) => failed.push((*pid, e)),
        }
    }
    nt_suspend_process(root_pid).ok();    // suspend root last

    SuspendResult { suspended, failed }
}
suspend_chain is ONLY allowed on Critical-severity rules — same gating as the other auto-actions. CI rejects on Medium/Low. The chain walk is bounded at 64 descendants to prevent runaway suspension during process-fork attacks.
